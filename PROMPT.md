# 解説作成プロンプト

このファイルは、新しい問題を `code-sensei` サイトに追加するときの構成・デザイン・書き方のガイドです。
`data/problems/` の JSON（参照用データ）と、実際に表示される `docs/problems/*.html`（手作りのゲーム風ページ）の
両方を作るときに参照してください。

---

## 対象読者

**小学生**（競プロ初心者）。以下を前提としない：

- 変数・関数の概念
- アルゴリズムの知識
- 数学記号（∑、∀、∃ など）
- `x` などの代数的な記号

---

## 書き方のルール

### やってはいけないこと

| NG | 理由 |
|----|------|
| 「最小ピースを x cm 以上にできるか？」 | `x` は小学生に伝わらない |
| 「答えを二分探索する」だけで終わる | なぜそれで答えが出るかが伝わらない |
| 「最小値の最大化」という言葉だけで説明する | 「なんで最小を最大にするの？」で詰まる |
| コードのコメントで「〇〇を計算する」とだけ書く | `map` や `int` の意味がわからない |

### やること

**1. 動機を先に説明する**

「なんでそれを求めるの？」を最初に答える。
例：「スコアを最大にする ＝ 一番ガッカリする人をなくす ＝ 公平に切る」

**2. `x` を使わず「目標」か具体的な数字を使う**

```
NG 「最小ピースを x cm 以上にできるか？」
OK 「全部のピースを 13cm 以上にできる？」と聞く
OK 「目標の長さ以上に全部できるか？」
```

**3. 必ず具体的な数字でステップを追う**

```
「全部 10cm 以上にできる？」→ できる
「全部 13cm 以上にできる？」→ できる
「全部 14cm 以上にできる？」→ できない
→ できる最大が 13 → 答えは 13
```

**4. 「なぜその変換で答えが出るのか」を必ず説明する**

「〇〇以上にできる？」という質問は「答えが〇〇以上か？」と同じ。
最後の「できる」= 答え。

**5. 「使わなかった場合」を具体的な失敗で示す**

「何回かかるか」「なぜ TLE になるか」を実際の数字で示す。

## 絵文字について（重要）

サイト内の見出し・ラベル・マスコットのセリフなどに、装飾目的の絵文字（❌✅🔑💡🎉✂⚡🍡🌳 など）は使わない。
「一気にAI生成っぽさが出る」という理由で禁止。

- 使ってよい記号：★（難易度バッジ）、✓ / ✗（CSSの `::before` アイコンや「8 ≥ 8 ✓」のような数式的な意味での使用）
- それ以外の絵文字は、見出しや文中の装飾として付けない

## サイト構成とビルド方法

### ビルド

```bash
cd src
go run .           # 通常ビルド（既存の docs/problems/*.html はスキップされる）
go run . -f        # 強制フルビルド ※ docs/problems/*.html の手作りページが消える！
go run . serve     # ローカルサーバー起動 (localhost:8080)
```

`data/` → `src/main.go` → `docs/` の流れで自動生成ページ（index / glossary / code_reading / style.css）が作られる。
`docs/problems/*.html` は手作りのため、`-f` を使うと消えるので必ず Git にコミットして保管すること。

### デザイン

- **配色**: ホワイトモード（背景 `#fff`、文字 `#111`）。ヘッダーは薄いグレー `#f5f5f5`。
- **アクセント色**: 青 `#1565c0`（リンク・見出し・選択中タブ）、グリーン `#2e7d32`（正解）、レッド `#c62828`（不正解）、オレンジ `#e65100`（強調）
- **「エディタ」領域だけダーク**: コード実行パネル（`.runbox` など）・アルゴリズムの状態表示（`.bs-state` 等）・コードブロックだけは黒背景 `#050505` の「コンソール」として残し、それ以外はすべて白背景にする
- **フォント**: DotGothic16（日本語対応ピクセルフォント） + Google Sans Code（コード部）
- **CRTエフェクト**: `body::after` に repeating-linear-gradient で薄いスキャンライン（`rgba(0,0,0,.03)` 程度、控えめに）
- **絵文字は使わない**（上記「絵文字について」参照）

### ページ種別

#### 1. 自動生成ページ（`src/main.go` がビルド時に生成、編集不要）

- `docs/index.html` — 問題一覧
- `docs/glossary/*.html` — アルゴリズム用語集
- `docs/code_reading.html` — プログラムの読み方
- `docs/style.css` — 共通スタイル（`css()` 関数が源泉）

印刷用ページ（`docs/print/*`）は生成しない（廃止済み）。

#### 2. 手動作成ページ（全画面スライドショー形式）

- `docs/problems/typical90_*.html` — 各問題の学習ページ

スライドショーページの構成（問題の内容に応じて6〜7画面）：

1. **S1 問題文** — 問題と入出力例、AtCoderリンク、関連する用語集リンク（`.ref-links`）・プログラムの読み方リンク
2. **S2 解説** — かんたん解説（子ども向け、具体的な数字で）。カードが3枚以上になる場合は
   一度に全部出さず、`.ecarousel`/`.estep` パターンで1枚ずつ送れるようにする（下記参照）
3. **S3 確認（正しいアルゴリズム）** — ステップ可視化のみ（コード実行ボックスは置かない。重複するため）
4. **S4 間違い（悪いアルゴリズム）** — 悪い例のコード + 「なぜ問題か」の説明のみ
5. **S5 速度比較**（悪いコードと正しいコードの両方を実行して速度差を見せる内容がある場合のみ追加）—
   間違い/正解タブに埋め込まず、独立したタブにする
6. **正解** — 正しいコード全文 + 自分で試せる実行ボックス
7. **体験（TRY IT）** — インタラクティブな体験コーナー。番号なしの "STAGE" 扱い

「S3で実行ボックス」「正解ページでも実行ボックス」のように同じ「正しいコードを動かす」体験を
2箇所に重複させない。実行して試す体験は基本1箇所（正解タブ）にまとめる。

スライドページは `go run . -f` で削除されるため、手動作成したページは
GitHubへコミットして保管すること。

#### 解説カルーセル（`.ecarousel` / `.estep`）

S2などでカードが複数ある場合、以下のCSS/JSパターンを使って1枚ずつ表示する：

```css
.ecarousel{display:grid}
.estep{grid-area:1/1;visibility:hidden;pointer-events:none}
.estep.active{visibility:visible;pointer-events:auto}
```

```js
var eStepIdx=0, E_STEPS=<カード枚数>;
function updateEStep(){
  document.querySelectorAll('.ecarousel .estep').forEach(function(el,i){el.classList.toggle('active',i===eStepIdx);});
  document.getElementById('ectr').textContent=(eStepIdx+1)+' / '+E_STEPS;
  document.getElementById('eprev').disabled=(eStepIdx===0);
  document.getElementById('enext').disabled=(eStepIdx===E_STEPS-1);
}
function eStep(d){eStepIdx=Math.max(0,Math.min(E_STEPS-1,eStepIdx+d));updateEStep();}
```

`grid-area:1/1` で全カードを同じマス目に重ねることで、コンテナの高さが一番大きいカードに
自動的に合わせられ、カードを切り替えても下のナビ矢印がガタつかない（重要）。

#### モバイル対応

2カラムのレイアウト（`.playout` / `.vlayout` / `.duallayout` や S6 の左右パネルなど）は、
`@media(max-width:700px)` ですべて `flex-direction:column` に縦積みする。固定幅のサイドバー
（`.psamp`・`.yokan-wrap` 相当・S6の左右パネルなど）は `width:100%` にする。インライン
`style="width:XXXpx"` で幅を指定している要素は、先にクラス化してからメディアクエリで
上書きできるようにする。固定ピクセル幅のグリッド可視化（行列など）は `overflow-x:auto` で
横スクロールできるようにする。

#### レイアウトの高さ変動に注意

ステップごとに内容の行数が変わる要素（`.bs-state` のようなアルゴリズム状態表示など）は、
`min-height` を固定するか `.ecarousel` と同じ grid-stack パターンを使い、下にあるナビ矢印の
位置がステップ送りのたびにガタつかないようにする。

## マスコット（ドット絵ロボット）

各問題ページ・共通ページ（index/glossary/code_reading）の右下に、10×10ドットのロボット
マスコットを常時表示する。新しい問題ページを作るときも同じ仕組みを丸ごとコピーして含める
（既存の `docs/problems/typical90_a.html` の `.mascot-wrap` 周辺のCSS/HTML/JSをそのまま流用）。

### 挙動

- 通常は画面右下に浮遊アニメーション（フロート）で待機
- クリックすると次のいずれかをランダムで実行：左ダッシュ／上ダッシュ／回転（スピン）／
  ダンス／豆知識を喋る
- ごく低確率（8%）で、画面中央にジャンプして大きく表示され、豆知識を喋ってからまた
  右下に戻る（`.mascot-wrap.center` クラスで実装）
- アイドル中も数十秒に一度、自発的に上記のどれかを行う
- 問題ページでは正解／不正解の判定時に `mascotSetMood('happy'|'sad', メッセージ)` を呼び、
  対応する表情・アニメーションとセリフを出す

### 豆知識バンク（`MASCOT_TRIVIA`）

学習の心構え・デバッグのコツ・アルゴリズム豆知識・競プロ雑学・応援メッセージなど、
100個程度のバリエーションを持たせる。特定の問題に依存しない、どのページでも使える
内容にする。新しい豆知識を追加する場合も、この方向性に合わせること。

## 問題ページの参照用JSONフィールド（`data/problems/typical90_x.json`）

`docs/problems/typical90_x.html` は `main.go` によるテンプレート生成ではなく手作りだが、
本文・制約・サンプル・模範解答の**下書き**として、以下のJSONフィールドを先に埋めておくと
スライドページを作りやすい。

1. **問題文 + 制約 + 入出力の例**（`statement` / `constraints` / `samples`）
2. **かんたん解説**（`statement_note`）← ここが一番重要。S2の解説カードの元ネタになる
3. **制約の読み方**（`constraints_note`）
4. **悪い例コード → 正しいコード**（`bad_solutions` → `solutions`）

`easy_explanation` と `explanation` は現在使っていない（省略可）。

---

## JSON フィールドの書き方

### `statement_note`（かんたん解説）← 最重要

- 「どんな問題？」から始める（入力例1を使ってもよいが必須ではない）
- 問題のやりたいことを具体的な数字で説明する
- 「なんで全部試すと遅いの？」→「どうすれば速くなるの？」の流れを入れる
- O 記法（O(N log N) など）は使わない。代わりに「〇〇回の計算」「コンピューターでも何時間も」などで表現する
- `x` などの代数記号は禁止。「目標の長さ」「全部〇cm 以上」のように具体化する

### `constraints_note`（制約の読み方）

- 各制約を「〇〇が最大〇〇個」など日本語で説明
- O 記法は使わない。「〇〇 × 〇〇 = 〇〇回になって間に合わない」と具体的な数字で書く
- 「だから〇〇という方法を使う」まで書く

### `bad_solutions`（悪い例）

- `label` は削除してよい（現在ページに表示しない）
- `code`：コメントで「何がまずいか」を具体的に書く（「L が最大10^9なのに…」）

### コードのコメント

- `input()` など基本的な関数にも説明を書く
- 「何をするか」より「なぜそうするか」を優先する
- 1行1行に丁寧にコメントをつける（初心者向け）

## 生成時のプロンプト例

```
以下の競プロ問題の解説 JSON を生成してください。

対象：小学生（競プロ初心者）
言語：Python（コードのコメントも日本語で丁寧に）

【問題文】
（問題文をここに貼る）

【制約】
（制約をここに貼る）

【入力例・出力例】
（サンプルをここに貼る）

【使うアルゴリズム】
（例：二分探索 + 貪欲法）

出力する JSON フィールド：
- statement_note（かんたん解説。最重要）
- constraints_note（制約の読み方。O記法禁止）
- bad_solutions（Python のみ）
- solutions（Python のみ、全行コメント付き）

※ easy_explanation・explanation・samples[].explanation は現在使わないので省略可。

ルール：
- 説明中に「x」などの代数記号を使わない
- 具体的な数字でステップを追う
- 「なんで全部試すと遅いの？」→「どうすれば速くなるの？」の流れを必ず入れる
- O 記法（O(N log N) など）は使わない。「〇〇回」「何時間もかかる」などで表現
- コードのコメントは input()・int()・split() など基本関数にも書く
- 絵文字は使わない（★・数式的な✓/✗を除く）
```
