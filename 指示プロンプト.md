# 指示プロンプト（このプロジェクトを進める際の要約指示）

このファイルは、仕様書（`仕様書_アルゴリズム学習ゲーム.md`）の内容を踏まえて実装や追加のデザイン作業をAI（Claudeなど）に依頼する際に、そのまま貼り付けて使える要約プロンプトです。詳細な仕様・未決事項は必ず仕様書本体も合わせて参照してください。

---

## プロジェクト概要

子供（プログラミング未経験の小学生）向けに、Pythonの文法とアルゴリズム的思考を、勉強っぽさゼロのパズル/RPGゲームを通じて学べるツールを作っている。最終目標はAtCoderの問題が解けるようになること。大人が遊んでも面白いことを目指す。副次的に英語学習も兼ねる（セリフは英日併記、日本語オンオフ切替可能）。

## コアメカニクス

- 島全体はコードでできた世界。地面の一部に歯抜け（未生成のバグ）があり、そこにパーツをはめ込むとコードが実行され、地形やギミックが実体化する
- ロボ（主人公、記憶喪失）はグリッド上をキーボード操作で1マスずつ移動し、初期ポケモンの「かいりき」のようにパーツを押して隣接する歯抜けにはめ込む
- 1ステージ＝1つの短いコード。パーツは選択肢式（自由入力なし）、いつでも押し戻して取り消せる
- ワールド（順次実行／if／for／変数…）ごとに複数ステージがあり、ワールド内は自由順、全クリアで次のワールドが解放される

## ビジュアルスタイル指示（最重要・モックアップで確定済み）

新しく画面や素材を作る際は、必ず以下のトーンに合わせること。

**配色**
- 肌色・暖色系のベージュ／クリーム色は使わない。寒色系グレーで統一する
- 基準色: 地色 `#e4e7ea` / インク（線・文字） `#1c1c1c` / ミッドグレー `#8a9098`

**フィールド（マップ）画面**
- ピクセルアート（ドット絵）の岩・茂み・木・看板などを、マス目グリッド（tableまたはCSS Grid）上に配置する
- オブジェクトは全て「1マス」基準のサイズに統一し、スケール感を揃える（極端に大小がバラつかないようにする）
- スタート地点からコードエリアの入口まで、道（パス用のマス）が視覚的に繋がるようにする。道が無いと「置いてあるだけ」感が出て探索感が失われるので必須
- UIのHUD文字（タイトル・ステージ番号）はドットフォント（`Press Start 2P`など）を使う

**コード（パズル）画面**
- コードはカードや箱で囲わない。フィールドの中の「荒れた土のパッチ」のような地面テクスチャの上に、文字が直接書かれているように見せる
- コードの文字は手書きクレヨン風フォント（`Walter Turncoat`など、Google Fonts）を使う。等幅フォントでの表示はNG（「もろ出しだが勉強っぽくしない」という方針に合わせ、手書きの質感を優先する）
- 左右の余白は詰めすぎず、ゆったり取る

**歯抜け・パーツの表現**
- 角丸四角形（rounded rect）で統一する
- 状態ごとの描き分け:
  - 未生成（歯抜け）: 点線の角丸枠のみ、塗りなし
  - 操作中（ロボが今いる歯抜け）: 太めの実線の角丸枠、うっすら明滅するアニメーション
  - はめ込み済み／落ちてるパーツ: 黒（インク色）で塗りつぶし、文字は地の色に反転
- 全ての枠線は、SVGフィルター（`feTurbulence` + `feDisplacementMap`）などを使い、定規で引いたような綺麗な直線ではなく手描きのガタつきを持たせる

**ロボの見た目**
- ドット絵ベースの頭でっかちシルエット（頭:体がおよそ2:1。横向きも頭を体より一回り幅広に）
- 線には上記と同じ手描きワブル効果をかけ、質感を他の要素と揃える
- 4方向のスプライト（正面＝下向き、後ろ＝上向き、横＝右向き、左向きは横向きの左右反転）を用意する
- 本体はインク黒ではなく**ミッドグレー（`#8a9098`）**で塗る。文字・パーツがインク黒なので、ロボも黒だと見分けづらい。目などのディテールのみインク黒

**アニメーション**
- 操作中の歯抜けは薄く明滅させる
- ロボが歯抜けに向かって小さく前後に揺れる「押し込み」アニメーションを入れ、実際に操作している感覚を出す

## 実装時の技術メモ

**技術スタック（確定・実装中）**
- フロントエンドは `apps/web`（Vite + React + TypeScript、pnpmワークスペース）。ゲームエンジン（Phaser等）は使わない — グリッド移動・箱押しパズル主体でスクロール負荷や大量スプライトの要求が低く、Canvas/WebGLエンジンはオーバースペックと判断。手描きワブル等のCSS/SVG表現との相性も素のDOMの方が良い
- スタイリングはプレーンCSS（コンポーネントごとにファイル分割）。Tailwind等は使わない
- バックエンド（Cloudflare Pages/Workers + D1、認証、進捗保存）は**未着手**。フロントの手触りを先に固める方針
- インフラはCloudflare一本化（Pages/Workers + D1）、無料枠・カード登録不要な構成
- 認証は名前＋PINの簡易ログイン、サインアップ機能は作らない

**実装でわかった注意点**
- CSSはコンポーネント単位でファイル分割していても**クラス名はグローバル**（CSS Modules等を使っていないため）。異なるコンポーネントのCSSで同じクラス名（例: `.token-layer`）を使うと後から読み込まれた方が勝ってスタイルが衝突する。実際にロボの座標がズレるバグが起きた。クラス名はコンポーネント固有にする
- 手描きワブル用のSVGフィルター（`feTurbulence`+`feDisplacementMap`）は、**正確なピクセル位置で動くインタラクティブ要素（ロボ・パーツ）のコンテナには絶対にかけない**。フィルターは要素とその子孫全体をまとめてラスタライズしてズラすため、位置計算とズレて端で飛び出て見える。装飾的な背景レイヤー（土のパッチの見た目など）だけに分離して適用する
- 箱押し（ソコバン）パズルで、パーツをステージの本当の外周マスまで押し込めると、ロボが反対側に回り込めず二度と引き戻せなくなる（詰み）。ギミック領域の外側に歩けるだけの余白を1マス設け、パーツはその外周には入れないようにする
- 固定で書かれたコード文字（`robot.move(` など）の上は、ロボ・押せるパーツともに歩ける／通過できるようにする（文字自体は動かせないが、通行は妨げない）
- HTMLモックアップにCSS Gridを使う場合、古いレンダリングエンジン（例: wkhtmltoimageのようなプレビュー/キャプチャツール）では正しく表示されないことがある。**実際のモダンブラウザで表示確認する前提であればCSS Gridで問題ない**が、静止画キャプチャが必要な場合は`<table>`ベースのレイアウトに切り替えると安全
- 同様に`inset`ショートハンドや`aspect-ratio`も古いレンダラーでは効かないことがあるため、キャプチャ検証が必要な場面では`top/left/right/bottom`を個別指定し、高さは固定pxで指定する
