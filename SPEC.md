# Code Sensei 仕様書

## 1. 概要

小学生向けのAtCoder(競プロ)解説教材サイト。競プロの問題を題材にプログラミングを学ぶ。
主人公(プレイヤーのアバター)がマップを冒険し、マスコットのロボットは**主人公をそばで見守り支える妖精ポジション**として学習をガイドする(昔のドット絵RPG(ドラゴンクエスト等)の相棒キャラのイメージ)。

本サイトは**AtCoderの提出・ジャッジ環境そのものを再現するものではない**。あくまで「その問題をどう解くか」を教科書的にじっくり学ぶ場であり、**最終的には本家AtCoder上で実際に問題を解けるようになること**をゴールとする。ブラウザ内コード実行(Pyodide)はサンプル入力を試す学習目的であり、AtCoderへの提出・自動採点機能ではない。

現状は静的HTML(Goツールで一部自動生成)+GitHub Pagesで、進捗やユーザーの概念は存在しない。
本仕様書は、これをCloudflare上のフルスタックアプリ(TypeScript統一)に作り変え、
個別ログイン・画面単位の進捗記録・管理画面からのコンテンツ管理・Gather.town風ワールドマップを
備えた「ゲームのように成長していく学習アプリ」にするための仕様。

Goツール・静的HTML生成の仕組みは廃止する。既存資産のうち引き継ぐのは`data/*.json`の
コンテンツ(問題文・制約・サンプル・解答コード・用語集など)のみ。

## 2. 対象ユーザー・利用形態

- 学習者(生徒): 個別アカウントでログインし、問題に取り組み、進捗が保存される。
- 管理者: 別ロールでログインし、問題・用語集・code_readingコンテンツの追加編集、生徒アカウントの作成を行う。
- **自己登録は不可。** 生徒アカウントは管理者が作成する(管理画面 or DB直接投入)。
- サイト全体がログイン必須(今の共有パスワードゲートと同じ思想を、個別ログインで置き換える)。

## 3. 全体アーキテクチャ

```
[React SPA (Vite) — Cloudflare Pages]
        │ HTTP (同一頂点ドメイン配下 /api/*)
        ▼
[Hono API — Cloudflare Workers]
        │
        ▼
[Cloudflare D1 (SQLite) — Drizzle ORM]
```

- フロントエンドとバックエンドは別プロジェクトとして分離。
- モノレポ構成(pnpm workspace): `apps/web`(フロント)、`apps/api`(バックエンド)、
  `packages/shared-types`(共有型)、`scripts/seed`(初期データ投入)。
- 認証はD1の`sessions`テーブルで管理するセッションCookie(httpOnly/secure/sameSite)。
  即時失効可能なようJWTではなくサーバー管理方式を採用。
  Pages/Workersを同一頂点ドメイン配下にまとめ、Cookieのcross-site制約を回避する。

## 4. 機能一覧

### 4.1 認証
- 生徒ログイン/ログアウト。**ユーザー名+パスワード方式**(メールアドレス不要)。子供の個人情報を最小限にし、パスワードを忘れた場合も管理画面から再設定するだけで完結させる(メール経由のリセットフローは持たない)。
- 管理者ログイン(別ロール)。
- 自己登録UIは提供しない。パスワードはWorkers対応のハッシュ方式(bcryptjs or WebCrypto PBKDF2)で保存。
- ログイン試行のレート制限(ブルートフォース対策)を設ける。
- **セッションは長期保持**(30〜90日程度)。子供が使う端末で毎回ログインし直す手間を避ける。明示的にログアウトした場合のみ失効。
- 同一端末を複数人で共有する場合は、使うたびに明示的にログアウト/ログインしてもらう運用を前提とする(キャラ選択的な切り替え画面は今回のスコープ外)。

### 4.2 ワールドマップ(Phaser)
- **Gather.town風の、自由に歩き回れる広場マップ**。矢印キーでキャラを移動、当たり判定、カメラ追従。広場自体は自由に歩けるが、各ステージ(問題)の入り口(扉/門)が個別に「未解放/解放/クリア済み」の見た目を持つ形にすることで、自由な移動とゲーム性を両立させる(未解放の扉は鍵・鎖などで視覚的に閉じている)。
- 問題選択は「クリックでノードを選ぶ」のではなく、**キャラを歩かせてステージの入り口に近づき進入する**(接近トリガーで入場プロンプト表示)。
- **ステージ解除はレベル帯(難易度帯)方式**: 難易度は**AtCoderの★スケール**(★1〜2=初心者向け、数字が大きいほど難しい、★8以上はトップレベル)をそのまま採用する。マップは★の範囲ごとに「部屋」に分かれ、**同じ部屋(★帯)内の問題は自由な順番で解ける**が、**次の部屋(より高い★帯)に進むにはプレイヤーレベルが一定以上必要**(4.5のレベル参照)。★スケールをそのまま使うことで、将来AtCoder本家に移行したときの難易度感覚がそのまま活きる。
- `required_level`(必要レベル)は**★から自動算出**する(例: `required_level = (difficulty-1)×2+1`)。管理者は問題ごとに★だけ入力すればよく、バランス調整は算出式の修正だけで全問題に一括反映できる。
- ロック/解除/クリア状態はDBに保存せず、`problems`テーブルのマップ配置情報(`map_x`/`map_y`/`map_order`/`required_level`)と`progress`・ユーザーレベルから`GET /api/map`がその都度算出する。
- 問題が増えるたびに広場に入り口を追加していく形で拡張する。広場が手狭になってきたら、複数エリア(広場が繋がったマップ、★帯ごとにエリア分割)に分割することも将来的に検討。
- **古いタブレット/Chromebook等での動作を想定し、軽量に作る**: Phaserの描画をWebGL専用にせず、Canvasレンダラーへのフォールバックを有効にする。アセット(タイル・スプライト)のファイルサイズも抑える。
- **アバターカスタマイズ**: 生徒は自分のキャラの見た目(肌色・髪型・服・アクセサリ等)をある程度選べる。設定は`users.avatar_config`(JSON)に保存し、マップ上のスプライトはこの設定から合成描画する。
- **マスコット妖精**もマップ上に常駐し、主人公の近くを浮遊してついてくる(4.5参照)。

**素材(アセット)方針**:
- マップタイル(地面・装飾・柵など)、プレイヤーのベーススプライト(4方向歩行、パーツ分割済みでアバターカスタムに対応できるもの)は、既製のフリー/CC0ピクセル素材(Kenney.nl、itch.ioのRPGアセットパック等)を利用する。ゼロから描き起こさない。
- ステージ入り口アイコン(未解放/解放/クリア済み)は簡易な自作 or 素材の加工で対応。
- マスコット妖精だけは、既存のロボットマスコットのテイストを引き継いだオリジナルのドット絵として新規に描き起こす(サイトの顔となるキャラクターのため)。

### 4.3 問題ページ

**学びの原則**: 問題ページは教科書として機能する。ただし読むだけで終わらせず、要所要所で**手を動かす小さな仕掛け**(クリックして予測する、値を入れて試す、など)を挟む。中心となる問いは一貫して「**正解は何か・間違いは何か・なぜそうなるのか**」の3点セットを学べること — 正解の解法だけを見せるのではなく、間違った解法(S4)と正解の解法(S6)を対比させ、その理由(なぜ遅い/なぜ動かないのか、なぜこの方法だと速い/正しいのか)を具体的な数字で説明する。この3点セットの学びを、S3(ステップを自分で確認する)やS7(実際に手を動かして試す)のような対話的な仕掛けで裏付ける。

- 1問題につき7つの「画面」(S1〜S7)で構成するスライドショー形式(既存踏襲):
  - S1 問題(問題文・制約・サンプル・関連用語リンク)
  - S2 解説(かんたん解説カルーセル) → クリア後に**小ボス1**の理解度チェック
  - S3 確認(アルゴリズムのステップ可視化。問題ごとに手作り)
  - S4 間違い(悪い解法とその理由) → クリア後に**小ボス2**の理解度チェック
  - S5 速度比較(悪い解法/良い解法の実行時間比較。任意)
  - S6 正解(模範解答+ブラウザ内コード実行) → クリア後に**小ボス3**の理解度チェック
  - S7 体験/TRY IT(自由演習。問題ごとに手作り) = **大ボス戦**
- コード実行は**Pyodide(ブラウザ内Python実行)を全ページでシングルトン化**して使用。現状はPythonのみ対応だが、将来の多言語対応を妨げないよう`language`列で拡張可能な設計にする。
- S3(アルゴリズム可視化)とS7(自由演習)は問題ごとに個別のロジック・UIが必要(汎用化しない)。
- S1/S2/S4/S6のテキスト・コード部分と、S5の比較ロジックは共通コンポーネント化する。

**ボス戦(理解度ゲーム化)**:
- 各問題の部屋には**小ボスが3体(S2/S4/S6にそれぞれ対応)+大ボス1体(S7に対応)**がいる。
  - **小ボス**: S2(解説)・S4(間違い)・S6(正解)を読み終えたタイミングで、内容理解を確認する短い確認テスト(選択式クイズ、1〜数問)が出題される。正解すると対応する小ボスにダメージ演出が入り、全問正解で小ボス撃破。誤答してもペナルティはなく、マスコット妖精がヒントを出して再挑戦できる(懲罰的にしない)。
  - **大ボス**: S7(体験/TRY IT)そのものが大ボス戦。実際に手を動かして問題を解く既存のベスポーク演習(例: typical90_aの切れ目クリック演習)の成否がそのまま大ボスへの攻撃になる。3体の小ボスを倒していないと大ボスの部屋には入れない(=S7に到達する前にS2/S4/S6のチェックを済ませる必要がある)。
  - 大ボスを倒す = **その問題を「クリア」**とする。この「クリア」がワールドマップの次ステージ解除条件(4.2)、および進捗上の完了シグナルになる。
- 小ボス・大ボスの戦闘UI自体は共通コンポーネント化(HPバー・ダメージ演出・撃破演出)。ボスの見た目(スプライト)は問題ごとに変える必要はなく、タグ/難易度によるバリエーション程度で十分(素材コストを抑える)。
- **正誤判定は必ずサーバー側で行う**: `POST /api/checkpoints/:questionId/answer`にはクライアントが選んだ選択肢の番号のみを送り、サーバー側で`correct_choice_index`と照合して正誤を決める(クライアントが`is_correct`を直接送る実装にしない。DevToolsでの偽装を防ぐため)。
- **当てずっぽう対策**: 選択肢の表示順は問題を出すたびにシャッフルする。連打で正解を引き当てられないよう、誤答後は簡単な待機・再挑戦導線(マスコット妖精のヒント表示など)を挟む。
- **大ボス撃破シグナルの共通インターフェース**: S7はベスポーク実装のため、各問題のS7コンポーネントは共通の合図(例: `reportBigBossDefeated(problemId)`)でシステムに撃破を通知する。新しい問題を追加するたびにこの合図の呼び出し方が変わらないよう、`<ProblemShell>`または専用フックとして最初から用意する。

### 4.4 進捗管理
- 「問題 × 画面(S1〜S7)」単位で進捗(閲覧位置のブックマーク)を記録(`not_started`/`in_progress`/`completed`)。画面遷移のたびにAPIへ記録し、再ログイン時に読んでいた場所を復元する。
- **問題の「クリア」は画面到達とは別に定義する**: 小ボス3体(S2/S4/S6の理解度チェック全問正解)+大ボス(S7のベスポーク演習成功)を倒した時点でクリアとする。単に全画面を閲覧しただけではクリア扱いにしない。
- 理解度チェックの正誤は問題ごと・設問ごとに記録し(D1参照)、再ログイン時に「どの小ボスを倒し済みか」を復元できるようにする。
- ワールドマップの解除判定(4.2)は、この「クリア」シグナルを元に算出する。XPバー・レベルアップ等の演出も同様にクリア実績から算出する。

### 4.5 ゲーム的演出

**RPG要素の役割分担**(それぞれ「クリアするほど育つ/集まる」で一貫させる。能力値による難易度変化は導入しない):

| 要素 | 役割 |
|---|---|
| レベル | 問題クリア(大ボス撃破)ごとに獲得したXPで上昇。**次の★帯(難易度帯)の部屋に進むための機能的なゲート**(4.2)を兼ねる。 |
| XP | 小ボス撃破・大ボス撃破で獲得。レベルの原資。 |
| コイン | 問題クリアで獲得する通貨。ストアで防具(見た目アイテム)と交換できる。XPとは別枠(XPはレベルの原資、コインは買い物用)。 |
| アイテム/防具 | 問題クリアの**直接報酬**、または**ストアでコイン購入**で入手する見た目アイテム。アバターに装備すると反映(帽子・マント・盾等)。**能力値には影響しない**。 |
| 鍵アイテム(将来拡張) | ボーナス/隠し問題部屋の解錠に使う特別アイテム。スキーマ上は対応可能な形にしておくが、初期実装では使わなくてよい。 |
| HP/スタミナ | **未定**。ペナルティなし方針(誤答で減らない)と素直に両立する役割がまだ定まっていないため、導入するかどうか含めて保留。 |

- XPバー・レベルアップ演出・マスコットのリアクションは、ゲームエンジンではなく通常のReact/CSS/canvasアニメーション(必要に応じてFramer Motion、canvas-confetti等)で実装する。
- マスコット(ピクセル絵ロボット)は「先生」ではなく、**主人公(プレイヤー)をそばで見守り支える妖精ポジション**。昔のドット絵RPG(ドラゴンクエスト等)の相棒キャラのようなイメージで、ワールドマップ上では主人公の近くを浮遊してついてきて、問題ページでは正解/不正解のリアクションや励まし・ヒントを出す役割。
- 全画面共通コンポーネント化し、`useMascot()`フックで気分表出(`happy`/`sad`等)を制御できるようにする。
- **絵文字は一切使用しない**(見出し・ラベル・マスコットのセリフを含む、全箇所で禁止)。

### 4.6 管理画面
- 管理者ロールでログインし、以下を管理:
  - 問題(問題文・制約・サンプル・模範解答/悪い解答のコード・タグ・難易度・用語集リンク・マップ配置)
  - S2/S4/S6の解説カード・箇条書き(構造化してDB管理、管理画面から編集可能)
  - S2/S4/S6の理解度チェック(小ボス)問題(選択式クイズ・正解・簡単な解説)
  - 用語集(glossary)
  - code_reading(基礎文法解説)
  - アイテム(見た目・価格・クリア報酬設定)
  - 生徒アカウントの作成
- 管理画面が扱うのは**テキスト/コードデータのみ**。S3の可視化・S7の演習など問題固有のベスポークな視覚要素は対象外(開発者がコードとして実装する)。
- **問題の削除は基本的に非公開化(`is_published=false`)で行う**。物理削除は生徒の進捗・理解度チェック結果もCASCADEで消えるため、誤操作防止のため別途確認つきの操作にする。

## 5. データモデル(D1 / Drizzle)

```
users(id, username unique, password_hash, display_name, role['student'|'admin'],
      avatar_config json /* 基本の見た目(肌色/髪型/服) */, xp, level, coins,
      created_at, updated_at)
sessions(id, user_id fk, expires_at, created_at)

items(id, name, description, icon_key, slot['hat'|'cape'|'shield'|'other'] null
      /* 装備可能ならスロット種別、コレクションのみならnull */,
      price_coins null /* nullなら非売品(クリア報酬専用)、値があればストアで購入可 */,
      is_key bool default false /* 将来のボーナス部屋解錠用、初期実装では未使用可 */)
user_items(id, user_id fk, item_id fk, source['clear_reward'|'store_purchase'],
           source_problem_id fk null /* クリア報酬の場合どの問題か */,
           is_equipped, acquired_at)  UNIQUE(user_id, item_id)

tags(id, name unique)
problems(id, contest, problem_number, title, atcoder_url,
         difficulty /* AtCoderの★スケールをそのまま採用(★1〜2=初心者、数字が大きいほど難しい) */,
         required_level /* この★帯の部屋に入るために必要なプレイヤーレベル */,
         statement_md, constraints_md, constraints_note_md, statement_note_md,
         map_x, map_y, map_order,
         clear_reward_item_id fk null /* クリア報酬アイテム */,
         is_published, added_at, created_at, updated_at)
problem_tags(problem_id fk, tag_id fk)

samples(id, problem_id fk, position, input, output, explanation_md)
solutions(id, problem_id fk, language, code, steps_json)  UNIQUE(problem_id, language)
bad_solutions(id, problem_id fk, language, label, code)   UNIQUE(problem_id, language)

explanation_cards(id, problem_id fk, screen['s2'|'s4'|'s6'], position, title, body_md, variant)

checkpoint_questions(id, problem_id fk, screen['s2'|'s4'|'s6'] /* 対応する小ボス */, position,
                     question_md, choices_json, correct_choice_index, explanation_md)
checkpoint_answers(id, user_id fk, checkpoint_question_id fk, is_correct, answered_at)
                   UNIQUE(user_id, checkpoint_question_id)

glossary_entries(id, name, short, description_md, without_label, without_code,
                  with_label, with_code, when_to_use_md)
problem_glossary_links(problem_id fk, glossary_id fk)

code_reading_entries(id, name, short, body_md, python_code, other_note)

progress(id, user_id fk, problem_id fk, screen['s1'..'s7'],
         status['not_started'|'in_progress'|'completed'], completed_at, updated_at)
         UNIQUE(user_id, problem_id, screen)
```

正規化方針:
- samples / solutions / bad_solutions は今のJSON({python:{...}})形ではなく`language`列を持つ子テーブル。管理画面での行単位CRUD・将来の言語追加を「行を足すだけ」にするため。
- `steps_json`はJSONのまま(現状未使用、独立編集の必要性が薄い)。
- `easy_explanation`/`explanation`(旧スキーマの未使用フィールド)は廃止。
- ワールドマップのノード状態(ロック/解除/クリア)は保存せず、`user.level`と`problem.required_level`の比較+進捗データから都度算出する(★帯をまたぐ場合のみレベル判定、同★帯内は常にオープン)。
- **「問題クリア」(大ボス撃破)も保存せず都度算出する**: 「S2/S4/S6の`checkpoint_questions`全問について`checkpoint_answers.is_correct=true`が揃っている(=小ボス3体撃破)」かつ「S7の演習成功シグナル(`progress`のS7行が`completed`)」が揃った状態を「クリア」とみなす。管理画面で問題ごとに理解度チェック問題数を自由に増減できるため、固定のクリアフラグを持たせるより都度判定の方が変更に強い。
- **`users.xp`/`level`/`coins`は例外的にストア(直接ミュータブルなカウンタ)として保存する**: 都度算出パターンと異なり、これらは単純な加算カウンタで、ゲート判定に使うのは`level`のみ(★帯解除)。クリア/ボス撃破イベントごとにAPI側でXP・コインを加算し、レベルはXPから式(例: `level = floor(xp / 100) + 1`)で計算・同期する。

## 6. API仕様(Hono)

| ルート | 認証 | 概要 |
|---|---|---|
| `POST /api/auth/login` / `logout` / `GET /api/auth/me` | — | セッションCookie発行/破棄 |
| `GET /api/problems`, `/:id` | 生徒ログイン | 問題文・サンプル・解答コード・理解度チェック問題等 |

## 8. デザイン方針

- **配色・レイアウト・CRTスキャンラインなどの見た目は旧サイトを踏襲せず、`code-adventure`として作り直す。**
- 維持するのはフォント(DotGothic16(和文ピクセルフォント)、Google Sans Code(コード))と、マスコットのドット絵の雰囲気のみ。
- **絵文字禁止**(AIっぽく見えるため、との既存方針)。
- モバイル対応(`@media(max-width:700px)`程度のブレークポイントでスタッキング)。
- 問題文中で説明する際は抽象的な変数(x等)ではなく具体的な数値で説明する、カード3枚以上はカルーセル化する、実行ボックスはS6のみに置く、といった`PROMPT.md`のペダゴジー方針は新実装でも踏襲する。

## 10. 移行方針

- 引き継ぐのは`data/*.json`のコンテンツのみ。Git管理のseedスクリプト(`scripts/seed/seed.ts`)で初回D1投入。
- 既存の手作りHTML(`docs/problems/*.html`)の実装はそのまま移植せず、内容を参考にしつつ新スタックで作り直す。
- 削除対象: `src/`(Goツール一式、コミット済みバイナリ含む)、`execution/`(未使用の実行スキャフォールド)、`docs/`(自動生成分)、`data/problems/typical90.json`(参照されていない古い重複ファイル)。

## 12. スコープ外(将来対応)

- Python以外の言語の実際の実行(スキーマ上は拡張可能な形にしておくが、実装は行わない)。
- 管理画面からのベスポークな視覚要素(S3可視化・S7演習)の作成。

- 複数人での端末共有時のキャラ選択的なログイン切り替えUI(都度ログイン/ログアウトで対応)。
