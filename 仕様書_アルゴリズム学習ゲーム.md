# 仕様書：アルゴリズム学習ゲーム（仮）v7

## 1. コンセプト

**「勉強」を一切感じさせない、パズルゲーム。**
遊んでいるうちに、気づいたらプログラミング的思考（順次・条件分岐・繰り返し・変数など）が身についている状態を目指す。子ども向けだが、**大人が遊んでも普通に面白い**ことを目指す（ストーリー・パズルの質で担保する）。

- 参考: *BABA IS YOU*（ルールそのものを書き換えると世界の法則が変わる、というメタな構造）／初期ポケモンなどのレトロRPG（グリッド上を移動し「かいりき」的に押してインタラクトする操作感）／カービィのような「ワールド選択→中に複数ステージ」という進行構造
  - ※特定作品のキャラクター・世界観・アセットを直接使うことはしない。あくまで**構造・操作感のみ**を参照する（非商用・個人利用であっても、既存キャラクターの流用は避ける）
- Pythonのコードは**隠さず、もろ出しにする**。ただし「勉強っぽさ」は出さない。この二つは矛盾しない、という前提で設計する
- NG: レッスン番号、解説文、「クイズ」「正解／不正解」といった学習アプリ的な語彙・UI・演出
- OK: ステージ、しかけ、ギミック、クリア、隠し要素、バグ、修復

最終ゴール（保護者側の目的）は「Pythonで AtCoder の問題が解けるようになること」。地面のコードが本物のPythonであるため、これは裏の目的ではなく、そのままゲームの中身になっている。実際に学習ツールとして継続利用したいため、**進捗管理は必須要件**。

**副次的な狙い（確定）**: セリフ・テキストを通じて英語学習も兼ねる。ゲーム内の会話・テキストは**すべて英語＋日本語併記**を基本とし、日本語表示は設定でオン／オフをすぐ切り替えられるようにする（英語のみでも遊べる状態を目指す）。

## 2. 対象ユーザー・プラットフォーム

- 小学生（プログラミング未経験）。ただし大人が遊んでも面白いレベルを目指す
- **キーボード操作前提。スマホ／タッチ操作は対象外**（レトロRPG的なグリッド移動・押し操作と相性が良いため）

## 3. 世界観・キャラクター

**世界設定: この島は、誰かが書いたコードでできている。**

- 地面や建造物、ギミックの実体は、地表に直書きされたPythonコードそのもの
- コードの一部が歯抜け（未生成）になっている場所があり、そこは半透明のノイズ／チラつきとして表示される。これは「世界がまだ生成しきれていないバグ」
- 歯抜けにパーツをはめ込むと、そのコードが実行され、対応する地形・ギミックが実体化する（例: `for i in range(3):` の `3` を埋めると、対応する歯車や扉が実際に3回動く仕掛けとして生成される）
- **コード＝物理法則が一致する**のが世界の核。書いたコードの通りに世界が動く

**主人公: ポンコツロボ**

- 記憶喪失の状態で目覚める。「この世界のバグを直しに来た」と思い込んでいる
- 自分では知識を持たず、フィールドで拾った部品（パーツ）を取り付けることで新しい動きを獲得していく（「賢くなる」のではなく「パーツが増える」）
- 表情はシンプルな目・耳パーツ程度。喋らず、鳴き声的な効果音とジェスチャーで感情を出す

**ビジュアルトーン**

- Pythonの文字（`for` `if` `range` など）はそのまま画面に出す。装飾でごまかさない
- 参照点は「ジョットの絵の中の文字の使われ方」＝文字が装飾ではなく世界に実在するモノとして描かれる感覚（碑文・銘板のような物理的な文字の存在感）。絵本的な世界観そのものを目指すわけではない
- レトロRPG（初期ポケモン等）のドット絵・グリッドタイルの質感を基調にする。「タイルの集合でできた世界」という画作りが、「この世界はコードでできている」という設定と自然に一致する

**ビジュアルスタイル（確定・モックアップ検証済み）**
- 配色は肌色系を避け、寒色系のグレー（例: `#e4e7ea`地・`#1c1c1c`インク・`#8a9098`ミッドグレー）を基調にする
- **フィールド（マップ）画面**: ピクセルアートのドット絵（岩・茂み・木・看板など）をマス目グリッド上に配置し、スタートからコードエリアの入口まで道（パス）がつながって見えるようにする。オブジェクトは全て1マス基準のサイズに統一し、スケール感を揃える。UIチロム（HUDのタイトル・ステージ番号）は `Press Start 2P` のようなドット文字フォントを使う
- **コード（パズル）画面**: コードは「地面の一角にある荒れた土のパッチ」のようなテクスチャの上に、カードや枠で囲わず**直接**書かれているように見せる。コードのフォントは手書きクレヨン風の `Walter Turncoat`（Google Fonts）を使用し、文字自体が地面に書き込まれた質感を出す
- **歯抜け・パーツの見た目**: 角丸四角形で表現し、状態によって以下のように描き分ける
  - 未生成（歯抜け）: 点線の角丸枠のみ（塗りなし）
  - 操作中（ロボが今いる歯抜け）: 太めの実線の角丸枠、うっすら明滅するアニメーション
  - はめ込み済み／落ちてるパーツ: 黒塗りの角丸四角＋文字は地の色（色反転）
  - すべての枠線は手描きのガタつき（SVGフィルターによるワブル効果）を加え、定規で引いたような綺麗な線にはしない
- **ロボの見た目**: ドット絵ベースの頭でっかちシルエット（頭:体がおよそ2:1、横向きも同様に頭を体より一回り幅広にする）で統一し、線には同じ手描きワブル効果をかけて質感を揃える。正面（下向き）・後ろ（上向き）・横向き（右、左は横のミラー反転）の4方向分のスプライトを用意する。**本体の塗り色はインク黒ではなくミッドグレー（`#8a9098`）にする**（固定コード文字・パーツがすべてインク黒なので、ロボまで黒だとフィールド上で埋もれて見分けづらいことが実装で判明したため）。目などのディテールのみインク黒にして文字側と統一感を持たせる
- **アニメーション**: 操作中の歯抜けは薄く明滅、ロボが歯抜けに向かって小さく前後に揺れる「押し込み」アニメーションを付け、実際に操作している感覚を出す

## 4. コアゲームプレイ

**操作方式（確定）**
- ロボは常にグリッド上の1マスに存在し、キーボードの方向入力で1マスずつ移動する（完全ターン制、1回の入力で1マスのみ移動）
- パーツは初期ポケモンの「かいりき」で岩を押す操作のように、ロボが隣接するパーツを押して隣のマスへ押し出す、という物理的な操作にする
- 押した先が歯抜けでない場合も、その場に仮置きできる（パーツはフィールド上を自由に押し動かせる。唯一の正解ルートに一直線ではなく、試行錯誤を前提とした箱押しパズル的な設計）
- 一度はめたパーツはいつでも押し戻して取り消せる（一発勝負にしない）。この「取り消し」は反対側からロボが押し戻す操作で実現するため、**パーツをステージの本当の外周マスまで押し込めてしまうと、ロボが回り込めず二度と引き戻せなくなる（詰み）**。ステージ設計上、実際のギミック領域の外側に歩けるだけの余白を1マス設け、パーツはその外周には入れないようにする（実装上の制約として確認済み）
- マウス・タッチ操作は考慮しない

**歯抜け・パーツの仕様（確定）**
- 歯抜けにはめるパーツは、最初からフィールドに用意された決まった選択肢のみ（自由入力はなし）。これにより正解判定はパーツの組み合わせが一致するかで機械的に行える

**グリッド・画面まわり（確定）**
- コードはグリッド上でトークン単位（パーツ／キーワードひとつひとつ）が1マスに対応する
- 1ステージ内の数行のコードは、コードエディタのように行ごとに改行して上から下へ並ぶ
- ギミック（歯車・扉など）は、コードが並ぶグリッドと同じ場所に共存する（コードの横や上下に配置され、フィールド全体が「コードでできた地形」として一体化して見える）
- 行番号・インデント（行頭の空白）もマスとして存在し、Pythonらしい見た目を忠実に再現する
- ステージに最初から書いてある固定部分（歯抜けではない`print`や括弧など）は背景の一部として描画され、押し動かせない。押し動かせる＝可変なのは歯抜けにはめるパーツのみ。ただし**その固定文字の上をロボやパーツが歩く／通り抜けることは可能**にする（文字自体は地面に書いてあるだけで物理的な壁ではないため。歩けないと、一度はめたパーツを反対側から押し戻して取り消すという操作が成立しなくなる場面が実装で見つかった）
- ロボ自身もコードが並ぶグリッドの中の1マスとして存在し、コード行の上を直接歩き回る（別レイヤーで外側から操作するのではない）
- ロボは常に画面中央（マスの中心）に位置し、ステージが画面に入りきらない場合はフィールド側がスクロールする
- パーツを歯抜けの隣まで押し、隣接した瞬間に自動ではまり込む（追加の確定操作は不要）
- ロボ自身がコードの一部（変数・引数など）として扱われるステージと、そうでない（純粋にパーツを押すだけの）ステージの**両方を用意する**（確定）。例えばロボの現在位置が`print()`の引数として扱われる、といったギミックを一部のステージに混ぜる

**画面UI・タイマー（確定）**
- グリッド以外の常設UIは最小限（位置情報など）に留め、画面のほとんどをグリッドが占める
- 制限時間・タイマーの類は一切設けない。パーツをはめるだけのシンプルなパズルに徹する
- 集めた友達は、ステージ選択画面などの一覧ではなく、**マップ上に実際に立っていて話しかけられる**形で確認する（NPCとして島に実在させる）
- 歯抜けを全部埋め終わった瞬間に、コードが一気に実行される（1行ずつ・都度反映ではない）

**友達（仲間集め）要素（確定）**
- 友達はロボットではなく、**各ワールドごとに異なるタイプの島の住人**として登場する（ワールドのコンセプトと絡めやすくするため）
- 友達は毎ステージ1人ずつ増えるわけではなく、**一部のステージにのみ**登場する（1ワールドのステージ数に対して一定割合、例：10ステージ中3つに友達がいる）
- 友達は、対応するステージをクリアした直後、その場（クリアしたステージの位置）に登場する
- マップ上の友達に話しかけると、会話（セリフ）に加えて、選択肢などの小さなインタラクションが発生する。この会話・選択肢は**任意**（無視してステージクリアには進める）
- 会話中の選択肢の選び方は蓄積され、**終盤のエンディング分岐（プレイの質）に関与する**（例：優しさポイント的な指標を内部で蓄積し、選び方の傾向でエンディングが変わる）
- 他の実在ユーザーと繋がるソーシャル機能ではなく、**一人用ゲーム内の収集要素**として実装する（インフラ・スコープを増やさないため）

**ステージ構造（確定）**
- **1ステージ＝1つの短いコード**。序盤は数行レベルのごく短いコードから始める
- 1ステージ内で歯抜けを埋めきると、そのコードが実行され、ギミックが完成してクリア
- 「まとめる（関数化）」ギミックは採用しない。同じコードを使い回す体験は狙わず、**ステージごとに違うアルゴリズム的発想に触れること**を優先する

**歯抜けの粒度（方針）**
- キーワード穴埋め（`___ i in range(5):` → `for`）を基本とし、序盤〜中盤の主軸にする
- 数値穴（`range(___)`）は「見て数えられる具体物の反映」（集めたカギの数がそのまま歯車の回転数になる、など）として早期から混ぜてよい
- `x = x + 1` のような抽象的な変数操作・計算ロジックは、キーワードの語彙が揃った後の中盤〜終盤に回す

**失敗演出（確定）**
- ペナルティ演出は作り込まない。単純に「ステージがクリアにならない＝次へ進めない」だけでよい

## 5. 全体構造（マップ／ワールド／ステージ）

**進行構造**
- 島全体は地続きの1つのマップ
- 島の中に、扉で区切られた**特色のあるワールド（エリア）**がいくつかある
- ワールドの入り口には扉があり、閉まっている。**ワールドは順番に解放される**
- **ワールド解放条件（確定・暫定）**: そのワールド内の全ステージをクリアすると次のワールドの扉が開く。初版はこれで作り、詰まりが多いようなら「一定割合クリアでOK」等の緩和を後日検討する
- 各ワールドの中には**複数の短いステージ（1ステージ＝1コード）**が存在し、**同じワールド内のステージは好きな順番に挑戦できる**
- **各ワールドのステージ数（暫定）**: 1ワールドあたり10ステージ程度

**ワールド構成案（対応するPython構文とコンセプト）**

| ワールド | 対応するPython構文 | コンセプト（世界観） |
|---|---|---|
| 1 | 順次実行（print, 代入） | まっすぐ歩けない壊れたロボの練習場 |
| 2 | if文（条件分岐） | センサーが誤作動する霧の谷（見える／見えないの判定） |
| 3 | for文（繰り返し） | 同じ動きを繰り返す歯車の遺跡 |
| 4 | 変数・カウンタ | 物を数えないと開かない蔵 |
| 5+ | 条件の組み合わせ・比較・複合パズル | AtCoder A/B問題の**出題タイプ**（合計を求める、条件分岐、最大値探索など）を参考にしたオリジナル問題。**初版には含めず、将来のアップデートで追加する想定**。AtCoderの実際の問題文をそのまま使うのは著作権上避け、慣れた子には外部リンクで本物のAtCoderへ送り出す形にする |

（旧案にあった「関数」ワールドは、まとめるギミックを採用しないため削除）

**各ワールドのギミック詳細（確定）**
- **ワールド1（順次実行、練習場）**: ゴールまでの道が壊れた部品（論理回路のようなもの）で寸断されている。歯抜けを正しいコードで埋めて部品を修理すると、その区間の道がつながる。1ステージは複数行（複数の方向・命令の組み合わせ）でゴールに至る構成にし、難易度を上げていく。
  - **移動のPython表現（確定・実装済み）**: `robot.move("right")` のような移動関数呼び出し形式を採用。歯抜けは引数の文字列（`"right"` `"down"` `"up"` `"left"`）
  - **コードとフィールドの連動（確定・実装済み）**: パズルで完成させるコードの行の並び（例: `"down"` → `"right"` → `"right"` → `"up"`）は、フィールド上で壊れた区間を実際に迂回するのに必要な方向の並びと完全に一致させる。正解すると、フィールドに戻った時点で壊れたマスが本当に通行可能な道に変わり、プレイヤーはコードで示した迂回ルートをそのまま手動で歩いて渡る。パズルを解いた瞬間に自動でロボが移動する演出ではない（ステージ1で実装・動作確認済み）
- **ワールド2（if、霧の谷）**: 霧そのものがマスごとに濃さを持ち、歯抜けに正しい条件式（比較演算子・数値など）をはめると、対応するマスの霧が晴れる。「見える／見えない」はマス単位の霧の濃淡で表現する
- **ワールド3（for、歯車の遺跡）**: 歯車が回転して扉が開くギミック。歯車の回転数が `range()` の数値と連動する（歯抜けに正しい数値をはめると、その数だけ歯車が回転し、回転数に応じて扉が開く）
- **ワールド4（変数・蔵）**: 拾ったカギの数をそのまま数える単純なカウンタ。集めたカギの数だけ蔵の扉が開く（増減を伴う複雑な値の管理は行わず、まずは「数える」という具体的な行為に留める）

## 6. ストーリー（大筋）

**核心設定**
- 記憶喪失のロボが「この世界のバグを直しに来た」と思い込んで目覚める
- 進めるうちに、実はこの世界（島）を最初に壊した張本人がロボ自身だったと判明する（直しているつもりの行為の意味が反転する）
- ループはプレイヤーが周回する形では扱わない。「島は同じ災害を何度も繰り返してきた」という**過去の記録・痕跡として発見していくもの**にする（プレイヤー自身は1周のみ）
- 友達はワールドごとに異なるタイプの島の住人として登場する（ロボットではない）。ステージクリアごとに友達が増えていくが、終盤で**友達は実は「島が元通りに直ってほしくない」と思っていた**と判明する。理由は悪意ではなく、「元に戻ったら自分たちの居場所がなくなってしまう」という寂しさ・不安
- **「直ってほしくない」の演出方針（確定）**: 友達は本音を直接には言わない。理由は**遠慮**——ロボが一生懸命頑張っている姿を応援したい気持ちが強く、本音を言い出せずにいる。中盤までは表情や仕草など間接的なサインのみを出し、終盤の決定的な場面で一番仲の良い友達（最初にできた友達など）がついに本音を漏らす。「なんで言ってくれなかったの」というロボの問いに対し、「頑張ってるあなたを見るのが嬉しかったから」と返す、優しさの裏返しとして描く
- 対立オチにはせず、**友達の痕跡を残したまま、一緒に新しい島の形を作る**という第三の道を終盤で選ぶ橋渡し構造にする
- 過去に何度も繰り返されてきた"直しては壊れる"という記録（ループの痕跡）は、実は今回出会った仲間たちを生み出す土台になっていた、と終盤で分かる。**「頑張ったことは無駄じゃなかった、しかもそれは一人の力じゃなく仲間のおかげだった」**という着地にする

**伝えたいメッセージ（確定）**
1. 友達を大事にしよう（仲間の意見を無視せず、一緒に答えを探す）
2. 努力は無駄にならない、頑張ったら報われる（過去の積み重ねが、見えない形で今の自分を支えている）

**章構成（10章、大筋のたたき台。各章の細部・NPC個別描写は今後詰める）**

| 章 | 概要 | 対応ワールド（目安） |
|---|---|---|
| 1 | 目覚め。記憶なし、「バグを直しに来た」という使命感だけがある | ワールド1 |
| 2 | 島のあちこちに"以前の姿"の痕跡。小さな違和感 | ワールド1〜2 |
| 3 | 同じ場所が何度も直され、壊されてきた記録を発見。「自分は何番目の修復者か」という問い | ワールド2〜3 |
| 4 | 記録上の「数」と実際の痕跡の数が食い違う。記憶にノイズが走り始める | ワールド3〜4 |
| 5 | 仲間が増え始める。少しずつ島の日常が賑やかになっていく | ワールド4 |

## 7. インフラ・技術スタック（確定）

**前提条件**
- 無料枠のみで運用（クレジットカード登録不要な構成を優先）
- なるべく低レイテンシ（過去にVercel+Render+Neonの構成でRenderのコールドスタート（無操作後の初回リクエストで約50秒）に苦しんだ経験があり、これを避けたい）
- 進捗管理のためDBは必須。学習ツールとして継続利用したいため

**構成: Cloudflare 一本化（Pages/Workers + D1）**
- フロントエンド・API・DBをすべてCloudflareに寄せる
- **フロント＋API**: Cloudflare Pages（Functions）
- **DB**: Cloudflare D1（SQLite互換）。無料枠: 月間約1.5億回の読み取り、約300万回の書き込み、5GBストレージ（変動あるため利用開始前に公式ページで要再確認）。クレジットカード登録不要。コールドスタートの概念がほぼなく、エッジで低レイテンシに応答

**認証方式（確定）**
- 新規サインアップ機能は作らない。ユーザーは事前に手動（DB直接投入 or 簡易スクリプト）で作成
- ログインは「名前を選んでPINコードを入力」程度のシンプルな方式（子ども向け、Googleログイン等の外部OAuthは対象外）
- 進捗（クリア済みステージ・解放したワールド）はD1に保存し、ユーザーごとに管理する

**言語表示（確定）**
- セリフ・UIテキストは英語＋日本語併記を基本とし、日本語のオン／オフを即座に切り替えられる設定を用意する
- テキストデータは1行ごとに `en` / `ja` を両方持つ構造にし、表示側で日本語の出し分けを行う（DBまたは静的データのどちらに持たせるかは今後検討）
- 難易度バランスとして、**最初は日本語表示ありが前提**。日本語オフは英語のみで進められる上級者向けのチャレンジ要素という位置づけにする

**D1テーブル設計（案）**

```sql
-- ユーザー（手動投入）
CREATE TABLE users (
  id INTEGER PRIMARY KEY,
  name TEXT NOT NULL,
  pin TEXT NOT NULL,          -- ハッシュ化して保存
  created_at TEXT DEFAULT (datetime('now'))
);

-- ワールド定義
CREATE TABLE worlds (
  id INTEGER PRIMARY KEY,
  key TEXT UNIQUE NOT NULL,    -- 'world_01' など
  name TEXT NOT NULL,
  order_index INTEGER NOT NULL
);

-- ステージ定義（1ステージ=1コード）
CREATE TABLE stages (
  id INTEGER PRIMARY KEY,
  world_id INTEGER NOT NULL REFERENCES worlds(id),
  key TEXT UNIQUE NOT NULL,
  title TEXT NOT NULL,
  code_template TEXT NOT NULL, -- 歯抜け込みのコード
  blanks_json TEXT NOT NULL,   -- 歯抜けの正解パターンをJSONで
  order_index INTEGER NOT NULL
);

-- クリア状況（進捗の本体）
CREATE TABLE user_stage_clears (
  user_id INTEGER NOT NULL REFERENCES users(id),
  stage_id INTEGER NOT NULL REFERENCES stages(id),
  cleared_at TEXT DEFAULT (datetime('now')),
  PRIMARY KEY (user_id, stage_id)
);
```

ワールドの解放状況は専用テーブルを持たず、「そのワールドの全ステージがuser_stage_clearsに揃っているか」をクエリで判定する（テーブルを増やさず常に正しい状態が導出できるため、将来ステージ数が変わっても壊れにくい）。
